# NextXR — Entity Behavior & Dynamics Analysis (How every entity behaves, and how to build it)

> Goal: understand exactly how the current code maps assets/entities to behavior,
> see the gap between "ontology-typed" and "actually behaves," and get a concrete,
> per-entity plan to write generative behavior models for **everything** in the
> Common Facilities Pack + base ontology — the way your space twin had a physics
> formula per telemetry, but now done universally through the ontology.

---

## 0. The one realization that reframes everything

There are **two completely different kinds of "behavior"**, and the current codebase
only has one of them. You are asking for the other one.

| | **Generative behavior (what you want / what the space twin had)** | **Detective behavior (what `behaviors/` currently is)** |
|---|---|---|
| Question it answers | "Given this entity's state, its inputs, and time — what telemetry does it *produce* right now?" | "Given a telemetry value — is it anomalous? emit a Finding." |
| Direction | **Produces** telemetry + evolves state | **Consumes** telemetry, produces Findings |
| Example | Server: `P = P_idle + (P_max−P_idle)·load^1.4 + PSU loss + noise` | `cfp.transformer_over_temp`: "if oil temp > 85°C for 2 min → critical Finding" |
| Where it lives now | **Nowhere.** Faked by `feed/simulate*.py` | `behaviors/registry.py` + `behaviors/hvac`, `behaviors/cfp` |
| Coupling | Must read inputs from connected entities (graph edges) | Independent; reads a setpoint at most |

### What the code actually does today (the mapping, precisely)

```
feed/simulate_cfp.py            behaviors/registry.py              graph/writer.py
─────────────────────           ─────────────────────             ──────────────
simulate_ups()  ─┐              registry.watching("cfp:upsSoC")    Finding node
simulate_xfmr() ─┤  TelemetrySample(signal,value) ─▶ behavior.evaluate() ─▶ writer.create
... 9 generators─┘   (scripted ramp+gauss noise)    (threshold/zscore/      (FLAGS edge)
                                                      physics-residual)
```

- **Telemetry is generated by 9 hardcoded scripts** (`simulate_ups`,
  `simulate_transformer_temp`, `simulate_smoke`, `simulate_door`,
  `simulate_tank_level`, `simulate_heartbeat`, `simulate_filter_dp`,
  `simulate_chiller_cop`, `simulate_vibration`). Each is a **canned profile**:
  `normal_value + rng.gauss(0, σ)` for N minutes, then a linear `min()`-clamped
  ramp to simulate a fault. **No physics. No relationships. No coupling.** The UPS
  doesn't drain because the mains actually failed — it drains because minute ≥ 15.
- **Behaviors only detect.** All 16 are `evaluate(sample, query) -> [Finding]`. The
  only thing they read from the graph is a scalar like `setpoint`
  (`tier_c_threshold`) — they never read *inputs from other entities*.
- **Entity state does not evolve from inputs.** A node's `status`/`setpoint` sit
  static in Neo4j; nothing computes "this server's outlet temp given its load and the
  CRAC's supply temp." The "physics" file (`tier_a_physics`) is a *residual detector*
  (compares observed vs an exponential-approach prediction), not a generator.

### So the gap is: **there is no entity-dynamics layer at all.**

The ontology gives every entity a *type, a place, relationships, observable
properties (with units), ratings, a state machine* — but nothing turns that into
*live, self-consistent, coupled telemetry with realistic fluctuation and loss*. That
layer is exactly what your space twin's per-telemetry formulas were. We need to build
it **generatively and universally**, keyed off the ontology.

## 1. What the ontology already gives us to build on (the hooks)

The good news: the ontology is already rich enough to drive a generative engine. Per
entity we can read, at runtime, via `SchemaService` / `GraphQuery`:

- **Observable properties + canonical units** — 21 CFP signals defined
  (`cfp:temperature` DEG_C, `cfp:activePower` kW, `cfp:electricCurrent` A,
  `cfp:voltage` V, `cfp:flowRate` L/s, `cfp:vibrationVelocity` mm/s, `cfp:energy`
  kWh, `cfp:fillLevel` %, `cfp:powerFactor`, `cfp:smokeObscuration`, `cfp:doorState`,
  `cfp:occupancyCount`, `cfp:illuminance` lux, `cfp:soundLevel` dB, etc.). These are
  the *output channels* of the generative models.
- **Ratings & condition** — `cfp:ratedCapacity`, `cfp:conditionIndex` [0..1],
  `cfp:criticality`, plus per-instance props like `setpoint`. These parameterize the
  ideal model and the degradation.
- **Topology relationships** — `nxr:feeds`/`fedBy`, `cfp:suppliesAirTo`, `cfp:serves`,
  `cfp:backsUp`, `cfp:controls`, `nxr:dependsOn`, `hvac:servesSpace`,
  `nxr:locatedAt`/`containedIn`. These define **who feeds whom** — the coupling graph
  the generative engine must walk to get each entity's *inputs*.
- **State machines** — `cfp:EquipmentStateMachine` (off↔running→degraded→fault→off),
  `cfp:ElevatorStateMachine`, `cfp:WorkOrderStateMachine`, `hvac:AirHandler`,
  `nxr:Incident`. These bound the discrete-mode part of behavior.
- **Sensor↔observable wiring** — `nxr:Sensor` requires `sosa:observes <property>`
  and `nxr:monitors <feature>`. A sensor's job is to *report* the state of the
  feature it monitors — so sensor telemetry should be **derived from the monitored
  entity/space**, not invented independently.

## 3. The architecture to add: a generative **Entity Dynamics** layer

This is a NEW subsystem (suggest `nextxr-ontology/dynamics/`), parallel to
`behaviors/`. It does not replace the detection registry — it *feeds* it. The
detection behaviors keep working unchanged; they just start watching **real,
coupled, fluctuating** telemetry instead of canned ramps.

### 3.1 Core contract

```python
# dynamics/model.py
@dataclass
class EntityContext:
    entity_id: str
    canonical_type: str            # ontology class IRI
    props: dict                    # graph node props (setpoint, ratedCapacity, conditionIndex, status, ...)
    inputs: dict[str, "EntityState"]  # upstream entities by role: {"fedBy": ..., "suppliesAirTo": ...}
    space: "EntityState | None"    # the zone/room it sits in (ambient conditions)
    t: float                       # sim time (seconds)
    dt: float                      # tick length
    rng: random.Random             # per-entity deterministic stream

@dataclass
class EntityState:
    status: str                    # off/running/degraded/fault
    signals: dict[str, float]      # latest produced telemetry, by signal IRI
    internal: dict                 # hidden state (SoC, dust_mass, bearing_wear, oil_temp, ...)

class DynamicsModel(ABC):
    models: list[str]              # canonical_type IRIs this model drives (subclass-aware)
    produces: list[str]            # observable property IRIs it emits
    consumes: list[str]            # input roles it needs (relationship predicates)

    def init_state(self, ctx) -> EntityState: ...
    def step(self, ctx, state) -> EntityState:   # advance dt, return new state + signals
        ...
```

Each model implements the **ideal equation + the real-world deviation layers**
(noise, efficiency loss, degradation, mode-dependent behavior). That separation is
the heart of "ideal vs reality":

```
output = ideal_model(inputs, params)
         × efficiency(load, condition, temp)      # losses (η < 1, part-load curves)
         + degradation(age, wear, fouling)        # slow drift over runtime
         + fluctuation(rng, σ)                     # fast stochastic noise / ripple
         + events(mode)                            # discrete fault injection / state changes
```

### 3.2 The tick engine (coupled, relationship-aware)

The crucial difference from today's independent scripts: entities are **coupled**, so
the engine resolves them in **dependency order each tick**.

```
DynamicsEngine.tick(dt):
  1. Load topology once: nodes + edges (feeds, suppliesAirTo, backsUp, controls,
     servesSpace, monitors, locatedAt) via GraphQuery.
  2. Topologically sort by "feeds/supplies/serves" so producers update before
     consumers (e.g. Chiller → AHU → Zone → Server inlet). Break cycles (e.g.
     server-heat ↔ zone-temp) with a 1-tick lag (use last tick's value) or 2-3
     fixed-point iterations.
  3. For each entity in order:
        ctx = build_context(entity, resolved upstream states, space state, t, dt)
        state = model.step(ctx, state)
  4. Emit telemetry: for each produced signal, push a TelemetrySample into the
     SAME FindingsLoop the feed uses → detection behaviors fire as before.
  5. Periodically persist evolving state to the graph via GraphWriter.update
     (status transitions go through the state-machine gate automatically).
```

Spaces (Zone/Room) get a **thermal/occupancy model** too — they're the shared medium
that couples HVAC, IT load, and occupancy.

### 3.3 Registry + ontology binding

Mirror `BehaviorRegistry`: a `DynamicsRegistry` maps `canonical_type` → `DynamicsModel`
(subclass-aware, like `resolve_label` walks `subClassOf`). A model registered for
`cfp:FacilityEquipment` is the default; more specific ones (`cfp:Chiller`,
`cfp:Server`) override. This means **you only write as many models as there are
distinct dynamics**, and every subclass inherits one — exactly the ontology's
extend-downward philosophy.

### 3.4 How sensors fit

Sensors are **derived observers**, not independent generators. A
`cfp:TemperatureSensor` that `nxr:monitors <Zone>` and `sosa:observes cfp:temperature`
reports `zone.signals[cfp:temperature] + sensor_noise + bias + occasional dropout`.
This guarantees self-consistency (the sensor agrees with the thing it measures) and
is where measurement realism (noise, drift, calibration error, stuck-at faults)
lives.

## 4. Per-entity generative models (the formulas — for everything in the code)

These are the models to implement. Each lists **state**, **inputs (graph roles)**,
**ideal equation**, and **reality layer**. Units are the ontology's canonical units.
`η` = efficiency, `~N(0,σ)` = Gaussian noise, `cond` = `conditionIndex∈[0,1]`.

### 4.1 Electrical chain (model producers before consumers)

**UtilityFeed**
- State: `available` (bool), `voltage`, `frequency`.
- Ideal: `voltage=400V`, `frequency=50Hz`, always available.
- Reality: `voltage = 400·(1 + ~N(0,0.01)) − sag(totalDownstreamLoad)`; rare outage
  events (Poisson, λ≈1/month) flip `available=false` → cascades to UPS/Generator.

**Transformer** (IEEE C57.91 thermal, simplified)
- Inputs: load current from downstream (`feeds` targets' total power).
- State: `oil_temp`, `winding_temp`, `load_current`.
- Ideal load loss: `P_loss = P_noload + P_load_rated·(I/I_rated)²` (copper ∝ I²).
- Top-oil first-order: `dθ_oil/dt = (θ_oil_ult − θ_oil)/τ`, with
  `θ_oil_ult = θ_ambient + Δθ_rated·((1+R·K²)/(1+R))^0.8`, `K=I/I_rated`, `τ≈3 h`.
- Reality: ambient from outdoor/room; cooling-stage step at high load; `cond` lowers
  efficiency (more loss → hotter); noise `~N(0,0.3°C)`. Feeds `cfp.transformer_over_temp`.

**UPS** (Peukert + aging)
- Inputs: `fedBy` UtilityFeed availability; downstream `load_kW`.
- State: `soc%`, `mode∈{online,battery,recharge}`, `battery_temp`, `cycle_count`.
- Online: `soc≈100`, `output=load/η_inv` (η_inv≈0.95), draws from mains.
- On mains loss → battery: `dSoC/dt = −100·(P_load)/(η·E_batt_kWh)` with Peukert
  derate at high load; `runtime ∝ E_batt/load^1.3`. Recharge ramps SoC back when mains
  returns.
- Reality: capacity fade `E_batt = E_rated·(1 − 0.0002·cycle_count)·temp_derate`;
  output voltage sag under load; `soc` noise. Feeds `cfp.ups_on_battery`.

**Generator** (diesel genset)
- Inputs: start command when UtilityFeed lost & ATS transfers.
- State: `fuel_level%`, `output_kW`, `frequency`, `coolant_temp`, `run_hours`.
- Fuel burn: `L/hr = a + b·load_kW` (a≈idle burn, b≈0.25 L/kWh); `dFuel/dt = −burn`.
- Startup transient: frequency overshoot then settle to 50Hz; output ramps to load.
- Reality: load-step frequency dip; coolant temp first-order rise; `cond` lowers
  efficiency. Feeds `cfp.generator_fuel_low`.

**EnergyMeter / Circuit / Panel / Switchgear** — **aggregators**: sum
`activePower`/`current` of downstream `feeds` targets; `energy += power·dt`;
`powerFactor` from load mix. Pure derived; no independent dynamics.

**SolarArray**
- Inputs: time-of-day irradiance profile + cloud noise; cell temp from ambient.
- Ideal: `P = P_rated·(G/1000)·(1 − γ·(T_cell−25))`, `γ≈0.004/°C`, `G` = irradiance.
- Reality: cloud transients (multiplicative noise), soiling derate over time.

**BatteryStorage** — like UPS but bidirectional: charges from solar/grid surplus,
discharges to load; `SoC` integrator with charge/discharge η≈0.9, C-rate limits.

### 4.2 HVAC chain (Chiller → Pump → AHU → Zone → Sensors)

**Chiller** (Carnot-bounded refrigeration)
- Inputs: cooling demand `Q_load` from AHUs it `feeds`; condenser water temp from
  CoolingTower.
- State: `cop`, `power_kW`, `chw_supply_temp`, `condenser_temp`, `load_fraction`.
- Ideal: `COP_carnot = T_evap/(T_cond − T_evap)` (Kelvin); `COP = η_2nd·COP_carnot`,
  `η_2nd≈0.5`. `P_elec = Q_load/COP`.
- Part-load: COP follows an IPLV curve (peaks ~50–70% load).
- Reality: condenser fouling lowers COP slowly: `COP·= (0.85 + 0.15·cond)`; ambient/
  condenser-temp swings; noise. Feeds `cfp.chiller_cop_baseline`.

**Pump / WaterPump** (affinity laws)
- Inputs: speed command (from control loop / VFD), system curve.
- Ideal: `Q ∝ N`, `H ∝ N²`, `P_shaft = ρ·g·Q·H/η_pump`, `P_elec = P_shaft/η_motor`.
- State: `flow`, `pressure`, `power`, `vibration`, `bearing_wear`.
- Reality: `bearing_wear += k·runHours·(1/cond)`; vibration baseline rises with wear
  + imbalance harmonics + `~N(0,0.2 mm/s)`. Feeds `cfp.pump_vibration_baseline`.

**CoolingTower** — rejects condenser heat evaporatively: `condenser_return_temp =
wet_bulb + approach`, approach widens with fouling; fan power ∝ speed³.

**AirHandlingUnit / PrecisionCooler / FanCoilUnit / VAVBox** (fan + coil)
- Inputs: `fedBy` Chiller chw temp; `suppliesAirTo` Zone return temp; fan speed cmd.
- Fan affinity: `flow ∝ N`, `ΔP ∝ N²`, `fan_power ∝ N³`.
- Coil: `Q_coil = ṁ_air·cp_air·(T_return − T_supply)`; `T_supply = setpoint` while the
  chiller can meet demand, else rises (capacity shortfall).
- State: `supply_temp`, `return_temp`, `fan_speed`, `power`, `filter_dust`.
- Reality: `T_supply` tracks setpoint with first-order lag + overshoot + noise; under
  fault (low chw, dirty coil) supply temp drifts up → fires `hvac.temp_*`. The
  AirFilter child accumulates dust: `ΔP = ΔP_clean + k·dust`, `dust += c·flow·dt` →
  `cfp.filter_clogged`.

**AirFilter** — `filterDeltaP` as above (dust-loading integrator).

**Boiler** — mirror of chiller for heating: `Q_heat = ṁ·cp·(T_out−T_in)`,
`fuel/gas ∝ Q_heat/η_combustion`, η drops with fouling; hot-water temp first-order to
setpoint.

### 4.3 Spaces — the coupling medium (model these or nothing couples)

**Zone / Room** (lumped thermal balance — *this is the key coupling node*)
- Inputs: `suppliesAirTo` from AHU/FCU (cooling), internal gains from Servers/people/
  lights it `contains`, conduction to outside/adjacent zones, `occupancyCount`.
- Ideal energy balance:
  `C_zone·dT/dt = Q_internal + Q_solar + Q_conduction − Q_hvac`
  where `Q_hvac = ṁ_supply·cp·(T_zone − T_supply)`, `Q_internal = Σ server_heat +
  occupants·100W + lights`.
- State: `temperature`, `relativeHumidity`, `co2`, `occupancyCount`.
- Reality: CO₂ rises with occupancy and decays with ventilation; humidity from
  latent loads; diurnal occupancy profile. **Every environmental sensor in the zone
  reads from here.**

### 4.4 IT / compute (system 9)

**Server**
- Inputs: workload profile (diurnal + noise); inlet temp from its Room/CRAC.
- State: `cpu_load∈[0,1]`, `power_kW`, `inlet_temp`, `outlet_temp`.
- Ideal: `P = P_idle + (P_max−P_idle)·load^1.4`; `Q_heat ≈ P` (≈100% to heat);
  `outlet = inlet + P/(ṁ_air·cp_air)`.
- Reality: PSU efficiency curve `η(load)`; voltage ripple; **thermal throttling**:
  if `inlet_temp` high, `load` capped and `P` drops (a real, visible coupling: AHU
  fault → room hot → server throttles). This is the kind of cross-entity realism your
  space twin had per subsystem.

**EdgeNode / NetworkSwitch / WirelessAP / VideoRecorder**
- `heartbeat` (1, with rare dropout events → `cfp.heartbeat_loss`), `cpu_load`
  (diurnal), `temperature` (∝ load + ambient), `throughput`/`clientCount` (diurnal +
  noise), `storageUtil` (monotonic creep for recorders).

### 4.5 Water (system 6)

**WaterTank** — `dV/dt = inflow(pump) − outflow(demand) − leak`; `fillLevel = V/V_max`.
Leak event adds an outflow term → tank drains → `cfp.tank_low_level`.
**WaterMeter** — `flowRate` from demand profile; continuous nonzero flow at night →
`cfp.continuous_flow_leak`. **LeakSensor** — derived from tank/pipe leak state →
`cfp.leak_detected`. **SumpPump** — level-triggered on/off cycling.

### 4.6 Fire & security (mostly event-driven, safety-critical)

**SmokeDetector/Heat/Flame/Aspirating** — baseline `obscuration ≈ 0.5% + ~N(0,0.1)`;
fire event = exponential rise → `cfp.smoke_alarm`. **FireAlarmPanel** — aggregates
detector states. **AccessDoor/Reader** — discrete Poisson badge events; forced-open /
repeated-deny event injection → `cfp.door_forced`/`cfp.repeated_deny`. **Camera /
Intrusion / Barrier** — discrete state + motion events. **Elevator/Escalator** —
queueing model: trips per hour (diurnal), `floorPosition`, `doorCycles++`,
occasional fault.

### 4.7 People / mobile / maintenance (process generators)

**Person/Visitor/Vehicle/AssetTracker** — arrival/departure (Poisson, diurnal),
position tracks (random-walk within geofence). **WorkOrder/MaintenanceTask/Inspection**
— lifecycle generators that walk the `WorkOrderStateMachine` and can be *spawned by
Findings* (closing the operations loop). **ConditionAssessment** — periodically writes
`conditionIndex` (which feeds back into every degradation term above — a real twin's
aging loop).

## 5. How "ideal vs real" is implemented universally (the reality layers)

Every model composes the same four-layer stack, so realism is consistent and
configurable:

1. **Ideal model** — the deterministic physics/engineering equation (above).
2. **Efficiency / part-load** — `η(load)`, COP/IPLV curves, PSU curves, motor curves.
   This is where "the desired amount of current won't flow / power isn't perfectly
   delivered" lives.
3. **Degradation** — slow drift over `runHours`/`cycle_count`, modulated by
   `conditionIndex` (fouling, wear, capacity fade, calibration drift). Persisted to
   the graph so it survives ticks and shows up in the twin's history.
4. **Stochastic + events** — fast noise (`~N(0,σ)`, ripple, jitter), plus discrete
   fault/mode events (mains loss, leak, fire, bearing failure) injected on a schedule
   or randomly. Events flip state-machine `status` (validated by the gate) and
   cascade through the topology.

Sensors add a 5th, measurement-only layer: bias, noise, quantization, dropout,
stuck-at — applied to the *true* value read from the monitored entity.

## 6. How to proceed — concrete build plan

**Step 0 — decide the boundary (recommended):** keep `behaviors/` (detection)
untouched; add `dynamics/` (generation). The dynamics engine emits `TelemetrySample`
into the **existing** `FindingsLoop`, so all 16 detectors keep working and now watch
realistic signals. Retire `feed/simulate*.py` gradually (or keep as a "scripted" mode).

**Step 1 — scaffolding (small, reusable):**
- `dynamics/model.py` — `DynamicsModel`, `EntityContext`, `EntityState`, `DynamicsRegistry`
  (subclass-aware resolution like `resolve_label`).
- `dynamics/engine.py` — `DynamicsEngine`: loads topology via `GraphQuery`, topo-sorts
  by feed/supply/serve edges, runs `step()` per entity per tick, emits samples,
  periodically persists state via `GraphWriter.update`.
- `dynamics/registry_build.py` — registers models per `canonical_type`.

**Step 2 — model the coupling medium first:** `ZoneThermalModel` /
`SpaceModel` (so HVAC↔IT↔occupancy actually couple). Without this, everything is
independent again.

**Step 3 — implement models producer→consumer:** UtilityFeed → Transformer → UPS/
Generator → (EnergyMeter aggregators) → Chiller → Pump → AHU/CRAC → Zone → Server →
