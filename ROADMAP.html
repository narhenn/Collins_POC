Platform Roadmap & Improvement Register

Where the platform stands after the 20-agent expansion, what's still weak, and every meaningful improvement ranked by impact and effort. Three tiers: build now, build next, build for production.

Compiled from the Agent System Build Reference v0.3 · Platform state as of June 2026

Where we stand now

The MVP's 6 agents have been expanded to the full 20-agent product across four teams. Every flow is functional in stub mode (no API key) and enhanced with real LLM reasoning when the OpenAI key is configured. Here's the honest status of each component.

Known weaknesses in what we built

Things that work but aren't great

Schema Mapper in stub mode — without the LLM, it passes through bundle templates verbatim. No intelligence, no free-text mapping. With the LLM active it works but doesn't handle complex spatial relationships ("AHU-01 serves Server Room 3 AND the corridor"). Needs a relationship inference step.

Operational agents are batch, not streaming — they trigger after the feed loop completes a full run (~30s), not per-finding. A production system should trigger diagnosis when findings accumulate past a threshold, not on a timer.

Behavior Modeler always defaults to Tier C — in stub mode every fault becomes a threshold rule. The LLM path does real classification, but the Tier A skeleton it produces is just comments — a human engineer still writes the actual physics.

Asset Curator uses a static catalogue — 14 generic assets hardcoded. A real system would search a 3D asset database or generate assets on-demand.

Tier 1 — High impact, buildable now

These can be built in hours to days. Each one either fixes a visible gap or adds a feature users would interact with daily.

Tier 1 Real Copilot with tools ~3 hours

Replace the rule-based Copilot.jsx with a real LLM agent that has callable tools: GraphQuery for "show me findings on AHU-01", SchemaService for "what types exist", feed stats for live data, and ChangeLog for audit history. User asks a question in natural language, the agent queries real data and answers with grounded facts.

This is the feature people would use every day. It turns the platform from "a dashboard you read" into "a system you talk to".

Files: agents/copilot_agent.py (new), agents/copilot_graph.py (new), server/agent_routes.py, panels/Copilot.jsx, api/client.js
Tier 1 Drag-and-drop Vision upload ~30 min

Wire a file picker / drag-and-drop zone in the Concierge chat panel. On drop, convert to base64 data URI, POST to /agents/twin/upload (already exists), and continue the conversation. The Vision Agent processes the image and the Schema Mapper builds entities from what it sees.

Going from "drop a floor plan" to "here's your twin" is the single biggest demo wow-factor. The backend is done — this is pure frontend work.

Files: panels/Concierge.jsx (add drop zone + file reader)
Tier 1 Real-time WebSocket event stream ~2 hours

Replace the SSE polling on the Dashboard and LiveOps panels with a WebSocket connection. When a Finding fires, the Dashboard updates instantly. When the Diagnosis Agent writes a hypothesis, the Agents panel shows it live. The event bus already publishes every mutation — just fan it out over a socket.

Makes the platform feel alive. Real-time updates during the feed demo are dramatically more compelling than 2.5-second polling.

Files: server/main.py (add WebSocket endpoint), hooks/useEventStream.js (swap SSE for WS), panels/Dashboard.jsx, panels/LiveOps.jsx
Tier 1 Claude Sonnet as primary LLM ~1 hour

Swap the gateway from OpenAI to Anthropic. The spec says Claude Sonnet primary, Llama 3 fallback. The gateway is a single-point abstraction — change gateway.py to use the Anthropic SDK. Claude is stronger at structured output and multi-step reasoning, which directly improves the Diagnosis Agent and Schema Mapper.

Better reasoning quality across all 20 agents with a single file change. Also aligns with the spec.

Files: agents/gateway.py (swap OpenAI client for Anthropic client)
Tier 1 Pack Marketplace with runtime install ~1 day

Make the Marketplace panel functional. Clicking "Install" loads a bundle from the registry into the live ontology using the existing _register_fragment() function. Clicking "Uninstall" removes it. Show installed packs with their entity counts and rule counts. The bundle loader and registry already support this — the gap is the UI wiring.

Closes the gap between "bundles exist" and "bundles are installable at runtime". Makes the platform self-service.

Files: panels/Marketplace.jsx, server/agent_routes.py (add install/uninstall endpoints), agents/registry.py
Tier 1 Streaming Diagnosis trigger ~2 hours

Instead of running the operational agents after the entire feed loop, trigger them when findings accumulate past a threshold (e.g., 3+ findings on the same entity within 5 minutes). Subscribe to the event bus for Finding-create events, buffer per entity, and kick off the operational graph when the threshold is met.

Diagnosis in 30 seconds instead of after the full feed run. Makes the demo flow feel immediate.

Files: server/main.py (add bus subscriber), agents/operational_graph.py

Tier 2 — Medium effort, real differentiation

Each of these takes days to a week. They transform the platform from "working demo" to "competitive product".

Tier 2 MQTT / OPC-UA telemetry adapter ~3 days

Replace the simulated feed with a real protocol adapter. Connect to a Mosquitto MQTT broker (or OPC-UA server) that publishes sensor readings. The adapter converts wire-format messages into TelemetrySample objects and feeds them to the BehaviorRegistry. Even a simulated MQTT source (publishing on a cron) proves the architecture works with real industrial protocols.

Bridges the gap between "demo" and "deployable". The Plugin Scaffolder already generates adapter boilerplate — use it as the starting point.

Files: feed/mqtt_adapter.py (new), server/main.py (add adapter option to feed loop), requirements.txt (add paho-mqtt)
Tier 2 Predictive behaviors (Tier-B ML) ~1 week

Add real machine-learning Tier-B behaviors beyond the z-score baseline. An IsolationForest anomaly detector that learns multivariate patterns across signals. A Prophet-based time-series forecaster that predicts Remaining Useful Life (RUL). These feed the Predict panel with real forecasts instead of mock data.

Tier-B is the intelligence layer the platform was designed around. Without real ML, the three-tier model (physics / statistical / rule) has a weak middle.

Files: behaviors/ml/isolation_forest.py (new), behaviors/ml/prophet_rul.py (new), panels/Predict.jsx, requirements.txt (add scikit-learn, prophet)
Tier 2 Compliance twin ~1 week

Model compliance standards (ISO 55000 for asset management, ISO 27001 for security, IEC 62443 for industrial cyber, CERT-In requirements) as Document entities in the graph. Link them to assets and processes via nxr:governedBy. The Compliance panel goes from mock to real — showing which assets are covered by which standards, with a coverage percentage and gap analysis.

Compliance is the #1 reason enterprises buy digital twins. Showing standards coverage on the dashboard turns a technical demo into a business case.

Files: packs/compliance/ (new pack), panels/Compliance.jsx, server/query_api.py (add compliance query)
Tier 2 Multi-tenant dashboard ~3 days

Add a split-screen or tabbed view where you monitor multiple twins simultaneously. Each tab is a different tenant with its own feed, findings, and risk score. The tenant isolation is already built into every query and every write — the gap is purely in the frontend layout.

Proves multi-tenancy at the UI level, not just the backend. Essential for any operator managing more than one facility.

Files: panels/Dashboard.jsx (add tab/split mode), context/TwinContext.jsx (support multiple active tenants)
Tier 2 Relationship inference in Schema Mapper ~3 days

Upgrade the Schema Mapper to infer spatial and functional relationships from natural language. "The chiller feeds cold water to AHU-01 which serves Server Room 3" should produce three entities with two typed relationships (nxr:feeds and hvac:servesSpace). Use the ontology's predicate vocabulary (domain/range constraints) to validate proposed relationships before the Validator sees them.

Unlocks the "describe your facility and get a connected graph" experience. Currently entities are created but relationships are limited to bundle templates.

Files: agents/twin_agents.py (upgrade schema_mapper), tools/schema_service.py (expose predicate vocabulary to the mapper)
Tier 2 Simulation panel (scenario twin forking) ~1 week

Make the Simulation panel real. Fork a twin into a "scenario" tenant (copy the graph), then replay the feed with modified parameters ("what if the setpoint was 18°C instead of 22°C?"). Compare findings between the real twin and the scenario. This is what-if analysis grounded in the actual behavior models.

Turns the platform from "monitoring" to "decision support". Operators can test changes before applying them.

Files: twins/service.py (add fork/clone), feed/simulate.py (parameterised replay), panels/Simulation.jsx

Tier 3 — Ambitious, production-grade

These are multi-week efforts that transform the platform into something deployable at enterprise scale.

Tier 3 Three.js 3D scene renderer ~2–3 weeks

Build a real 3D visualisation panel using Three.js or Babylon.js. When a twin is committed, the Scene Generator produces a glTF scene descriptor. The frontend renders it as an interactive 3D view: click an asset to see its properties, findings, and status. Color-code by severity (green/amber/red). Animate data flow along nxr:feeds relationships.

Requires: a procedural placement algorithm (force-directed or grid-based), a glTF asset library (generic shapes first, domain-specific later), and a camera/navigation system. The Scene Generator stub already extracts the topology — it needs to produce actual vertex data.

This is what makes a "digital twin" feel like a digital twin. Without 3D, the platform is a smart database with a good UI. With it, operators see their facility.

Files: panels/SceneView.jsx (new), agents/twin_agents.py (upgrade scene_generator), packs/assets/ (glTF models), package.json (add three)
Tier 3 Temporal workflow engine ~1 week

Replace the SQLite checkpointer with Temporal. Every agent run becomes a durable workflow. A 30-minute interactive twin-building session survives a worker crash and resumes at the last completed step. The engine is already designed for this swap — the CompiledGraph interface stays the same; only the checkpointer backend changes.

Also enables: workflow visibility (see which step each session is on), automatic retries with backoff, and workflow-level timeouts.

Production systems crash. Without durability, a 25-minute twin-building session is lost to a restart. Temporal makes agent flows enterprise-grade.

Files: agents/engine.py (add TemporalSaver), docker-compose.yml (add Temporal server), requirements.txt (add temporalio)
Tier 3 GIS integration for geographic rendering ~2 weeks

Use Mapbox GL or CesiumJS to render facility layouts on actual geographic coordinates. The CFP ontology already declares latitude, longitude, and areaM2 as datatype properties. A map view showing your facilities with real-time status overlays (green/amber/red per site), click-to-zoom into a facility's 3D scene, and geographic clustering for fleet-wide views.

Enterprise customers manage dozens of facilities across countries. A map view is how they think about their estate.

Files: panels/MapView.jsx (new), package.json (add mapbox-gl or cesium)
Tier 3 Edge deployment runtime ~3 weeks

Package the BehaviorRegistry + Tier-C rules into a lightweight edge runtime that runs on industrial hardware (Raspberry Pi, Advantech gateway, Siemens IPC). The edge runtime evaluates rules locally with sub-second latency and publishes findings back to the cloud platform. The Rule Author already tags rules for edge vs cloud placement — the gap is the deployment packaging and the edge-to-cloud sync protocol.

Safety-critical rules must execute at the edge (no cloud dependency). This is a hard requirement for industrial deployments.

Files: edge/ (new package), edge/runtime.py, edge/sync.py, Dockerfile.edge
Tier 3 Authentication, RBAC, and audit ~2 weeks

Add proper authentication (JWT or OAuth2) and role-based access control. Four roles: Operator (view dashboards + start feed), Engineer (build twins + manage assets), Expert (author bundles + approve), Admin (platform extension + user management). Every API call is authenticated and every mutation is tagged with the real actor ID in the Change Log. The Plugin Scaffolder already generates auth middleware boilerplate.

No enterprise deploys a platform without auth. This is a prerequisite for any production pilot.

Files: server/auth.py (upgrade), server/main.py (mount middleware), frontend/src/context/AuthContext.jsx (new)
Tier 3 LangGraph migration ~3 days

Replace the in-house engine.py with the real LangGraph library. The engine was designed for this — same API surface (StateGraph, END, add_node, add_conditional_edges). Benefits: LangGraph Studio for visual debugging, LangSmith for tracing, built-in streaming, and community support. The migration is a swap of one import line per graph file.

Unlocks the LangGraph ecosystem: visual debugging, distributed execution, and production monitoring.

Files: agents/twin_graph.py, agents/bundle_graph.py, agents/operational_graph.py, agents/plugin_graph.py, agents/accelerator_graph.py (change import), requirements.txt (add langgraph)

If you build 3 things next

The recommended next sprint

  1. Real Copilot with tools — highest daily-use value, proves LLM integration depth, ~3 hours. Users will interact with this more than any other feature.
  2. Drag-and-drop Vision upload — massive demo impact, 30 minutes of frontend work. "Drop a floor plan, get a twin" is the pitch in one sentence.
  3. Three.js scene renderer — transforms the product from "smart database" to "digital twin" visually. The longest build but the biggest differentiator. Start with generic cubes color-coded by type; refine to real models later.

These three cover: intelligence (Copilot), onboarding (Vision), and visualisation (3D). Together they make the demo tell a complete story.
