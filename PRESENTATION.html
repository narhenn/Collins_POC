The Problem & The Vision

No screen yet. This is the pitch. Set the context before showing anything.

What we built is a universal digital twin platform. Not a digital twin for one building or one factory — a platform that can create a digital twin for any industry. HVAC, maritime, energy, data centres, hospitals. The key word is universal.

The problem with digital twins today is they're all custom. Every deployment is a bespoke engineering project. You need domain experts, you need months of integration, and you end up with something that only works for that one site.

We solved that with three ideas:

First, a formal ontology — 113 classes that describe the physical world. Equipment, sensors, spaces, findings, incidents, actions. Every entity in the system is governed by this ontology. Nothing enters the graph without passing a SHACL validation gate. This is what makes it universal — the ontology is the common language across all industries.

Second, capability bundles — plug-and-play industry packs. The HVAC pack adds air handlers and temperature rules. The Common Facilities Pack adds 82 classes for power, fire, security, water, and more. And here's the key: these packs can be authored by conversation. A domain expert talks to our AI, and a new industry vertical gets created. No code.

Third, an AI agent system. 20 agents across 4 teams that handle everything from onboarding a new twin to diagnosing an incident to scaffolding a platform extension. Every agent has a deterministic fallback, so the system works even without the AI — the AI makes it smarter, not dependent.
Pause. Let this land. Then say: "Let me show you."

The Closed Loop — Author a Bundle, Build a Twin

This is the star demo. It proves the platform is self-extending.

Show on screen
Open the app at http://localhost:8000
Navigate to Bundle Author in the sidebar
Type domain: refrigeration, bundle name: Fridge Pack, click Start
In the chat, type: cold storage facility with compressor units, temperature sensors, condenser coils. main fault is overheat when compressor fails
Watch the 9-stage pipeline rail light up as each agent runs
Point out the artifacts: ontology fragment (Turtle RDF), behavior models (Tier classification), rules (threshold), elicitation questions, asset manifest
Click Approve & Publish when the human gate appears
What you just saw is a domain expert — not a developer, not an ontologist — creating an entirely new industry vertical by having a conversation. Behind that conversation, 9 AI agents ran in sequence:

The Interviewer extracted the entity catalogue. The Ontology Drafter wrote formal RDF. The Behavior Modeler classified the fault as Tier C. The Rule Author produced a threshold rule. The Elicitation Designer generated questions that the twin-building concierge will use for this domain in the future. The Asset Curator checked our 3D library. The Linter validated everything against the platform's freeze rules. And the Publisher signed and registered the bundle.

That bundle is now live in the registry. Watch what happens next.
Now show the closed loop
Navigate to Build a Twin
Type: build me a refrigeration monitoring twin for my cold storage warehouse
Watch: Concierge → Classifier picks "refrigeration" → Composer loads the Fridge Pack you just published → Schema Mapper → Validator → Graph Writer commits
The Concierge understood the request. The Domain Classifier identified "refrigeration" and matched it to the bundle we authored 30 seconds ago. The Capability Composer loaded that exact bundle. The Schema Mapper converted it to ontology entities. The Validator ran SHACL checks. The Graph Writer committed it to Neo4j.

The bundle we authored is now a live twin. Step 1 created the vertical. Step 2 consumed it. That's the closed loop — and it's what makes this a platform, not a product.

The Living Twin — Dashboard, Feed, Real-Time Findings

Show the twin is alive and producing operational intelligence.

Show on screen
Navigate to Dashboard
Click Start Feed (top right area)
Watch: entity count grows, findings appear, risk score climbs
Point out the severity breakdown (critical / warning / info)
Navigate to Live Ops — show the real-time event stream
Navigate to Asset Graph — show the entities in the graph
The feed is simulating telemetry from the twin — temperature readings every half second. The platform's Behavior Registry is evaluating every sample against three tiers of anomaly detection:

Tier C — hand-written threshold rules. The one our bundle authored fires when temperature exceeds the setpoint by 3 degrees for 3 minutes.

Tier B — statistical baselines. A z-score model that learns what "normal" looks like and flags deviations.

Tier A — physics models. A first-principles thermal equation that detects when observed temperature violates the energy balance.

These three tiers work together. A threshold rule catches the obvious. The statistical baseline catches drift. The physics model catches what neither of them can see. And every finding is written through the single write path — SHACL validated, change-logged, and published to the event bus.

AI Diagnosis & Recommendations

Show the LLM-powered operational intelligence layer.

Show on screen
Navigate to AI Agents
Show the diagnosis cards with root-cause hypotheses and confidence scores
Show recommendations tagged auto vs manual
Navigate to Change Log — show the Diagnosis and Recommendation nodes in the audit trail
When findings accumulate, the platform groups them into an Incident. Then our Diagnosis Agent runs — an LLM that reasons over the finding patterns, the graph topology (what's downstream of the affected equipment), and past incidents. It produces ranked root-cause hypotheses with confidence scores.

Then the Recommender Agent takes that diagnosis and produces actionable recommendations. Each one is tagged "auto" — the system can handle it — or "manual" — a human needs to act. And there's an advisory-only lock: this agent can never actuate, only recommend. Safety-critical decisions stay with humans.

This entire chain — finding, incident, diagnosis, recommendation, action — is written to the graph through the same validated write path. It's auditable, it's traceable, it's tamper-evident. Every decision the AI made is in the change log.

Platform Extension — Plugin SDK & Accelerator Packs

Show that the platform is extensible by engineers, not just domain experts.

Show on screen
Navigate to Plugin SDK
Click Start, type: I want a behavior plugin for vibration monitoring on pumps
Show the generated Python code with the Behavior base class contract
Navigate to Accelerators
Click Start, type: build an HVAC accelerator pack for commercial buildings
Show the assembled manifest with bundles, adapters, and compliance docs
The Bundle Author lets domain experts create verticals. But what about platform engineers who want to extend the platform itself?

The Plugin Scaffolder generates SDK boilerplate for six extension points: data adapters, behavior models, visualization panels, webhooks, data transforms, and auth middleware. An engineer says what they want, and they get a code skeleton that follows the platform's contracts. They write the logic; the platform provides the structure.

The Accelerator Pack Composer goes one level higher. It assembles multiple bundles, adapters, and compliance documents into a Solution Accelerator Pack — like a pre-packaged "Maritime SAP" or "HVAC SAP" that a consortium lead drops into a bid. It selects relevant bundles from the registry, identifies adapter needs, and produces a tender-ready manifest.

The Architecture — Why This Works

Pull back from features to the engineering principles. This is what makes it credible.

Show on screen
Navigate to Change Log — show hash-chained entries
Open the Neo4j browser at http://localhost:7474 — run MATCH (n) RETURN n LIMIT 50 to show the graph
Show http://localhost:8000/api/v1/schema/types in the browser — the full type vocabulary
Show http://localhost:8000/api/v1/health — healthy status with Neo4j + Redis
Five disciplines hold this together:

One: Single write path. Only the Graph Writer mutates Neo4j. Every mutation passes through a SHACL validation gate. No agent, no API endpoint, no UI action can bypass it. This is why the change log is a faithful record — because there's only one door.

Two: Ontology governance. 113 classes across a formal ontology grounded in BFO and SOSA — ISO and W3C standards. Ten closed taxonomy categories, machine-enforced. You literally cannot add an 11th category without failing the governance check. The ontology isn't documentation; it's a runtime contract.

Three: State, not calls. Agents never call each other. They read and write a shared state object, and the graph decides who runs next. This means you can add, remove, or reorder agents without touching any other agent's code.

Four: Tenant isolation. Every entity, every finding, every event, every bus stream is scoped to a tenant. Multi-twin on a single platform instance.

Five: Graceful degradation. No API key? Agents use deterministic stubs. Neo4j down? The app loads in degraded mode. Redis down? In-memory event bus. Every external dependency has a fallback.

What's Next — The Roadmap

End with the forward view. Show you know what's missing and have a plan.

What we've built is the complete agent system — all 20 agents, all 4 teams, the closed loop, the operational intelligence layer, and the platform extension tools. But we know what's next:

Near-term — a real Copilot that operators can talk to, with tools that query the live graph. Drag-and-drop floor plan upload so the Vision Agent can extract assets from images. Real-time WebSocket streaming instead of polling.

Medium-term — MQTT and OPC-UA adapters for real industrial protocols. Machine-learning Tier-B behaviors for predictive maintenance. Compliance modelling for ISO 55000 and IEC 62443. What-if simulation by forking twins into scenarios.

Long-term — a Three.js 3D renderer so operators see their facility, not just data about it. Temporal workflows for crash-resilient agent sessions. Edge deployment for safety-critical rules. And full authentication with role-based access.

The foundation is in place. The ontology governs. The agents reason. The write path is auditable. Everything from here is additive — we extend the platform, we don't rebuild it.
End with: "Happy to take questions."

If the boss asks tough questions

QuestionAnswer
"What if the AI is wrong?"Every agent has a deterministic fallback. The system works without AI; the AI makes it smarter. Safety-critical decisions (bundle publish, rule authoring) always require human approval. The advisory-only lock prevents the Recommender from ever actuating.
"How is this different from existing digital twins?"Existing twins are bespoke. Ours is a platform — new industries are added by conversation, not code. The closed loop (author a bundle → build a twin from it) is the proof.
"What about security / compliance?"Every mutation is SHACL-validated and hash-chain logged. Tenant isolation is enforced at every layer. RBAC is on the roadmap. The audit trail is tamper-evident by design.
"Can this scale?"Neo4j handles millions of nodes. Redis Streams handle thousands of events/sec. The checkpointer can swap to Temporal for crash-resilient workflows. The architecture is designed for it; the current deployment is a demo.
"What's left to build?"3D visualisation (Three.js), real telemetry adapters (MQTT/OPC-UA), predictive ML behaviors, compliance modelling, and auth. All additive — the foundation doesn't change.
